## P0：販売前に必須（未対応）

### P0-1【最重要】テナント分離 → ✅ クライアント側は対応済み（b3e030e）※ルールの本番反映が残る
**やったこと**（サンプルデータのため移行不要と確認できたので直接転換）：既定を**端末内のみ（local-only）**に変更。
新規ユーザーは匿名認証せず、子どものPIIを一切送信しない（`cloud-sync.js` の `_decideAuthAction`）。クラウド同期は
`accounts/{uid}`（メール＋パスワード・所有者のみ）を推奨パスに昇格。家族コード(families)は明示オプトインの
レガシー（非推奨・設定の折りたたみ内）に降格。既定では誰も families を作らないため、コード推測による露出は
実運用上ほぼ消失。grow-onlyの単一doc肥大も既定では発生しない。

**あなたに残る作業（必須）**：
1. **Firebase Consoleで `firestore.rules` を反映**（サンドボックスから `firebase login` 不可＝手作業）。ルール
   ファイルはコメントを新モデルに更新済み。`accounts/{uid}` は既に owner-only で正しい。
2. **推奨：家族コード機能の完全撤去**。誰も使わないなら `firestore.rules` の `families` ブロックを削除
   （デフォルト拒否）し、設定UIのレガシー折りたたみ（アカウント設定内の `<details>`）と
   `cloudFamilySet`/`bootSyncFamily` を撤去。これで「コードを知れば読める」残存リスクも消える。
   その際、int系7テスト（`family:'0000'`で family同期を検証）は account モードへ書き換えるか撤去。
3. ルール反映後、匿名で `families/0000` が読めないことをREST APIで実地検証（過去のセキュリティ修正と同手順）。

**⚠️ UX影響**：既定がlocal-onlyになったため、家族ランキング／きょうだい対戦／きょうの家族もんだいは
**アカウント作成後（または家族コード設定後）にのみ**動作する。あなた自身の既存インストールも次回起動で
local-onlyに切り替わる（サンプルデータのため問題ないはず。同期を戻すには設定でアカウント作成）。

⚠️ **私が単独で「accounts既定化＋ルール封鎖」をしなかった理由**：オンボーディングにメール登録必須化という大きなUX変更を伴い、かつ既存の家族データを巻き込む後戻り困難な操作で、あなたの判断とConsole操作が要るため。移行UIの設計から一緒に進めるのが安全。

## P1：審査・権利・安全で高確率に問題化

### セキュリティ（コードで対応可能・次セッション候補）
- `escapeHtml`が引用符（`"`）を非エスケープ→同期される `voice_*` 設定値経由でストアドXSS。`js/util.js` の `escapeHtml` に `"`/`'` を追加し、属性値を使う箇所を確認。
- 管理者PIN/家族PINが `btoa`（暗号化でない・共有同期で復元可能）。子どもの誤操作抑止としては可だが「保護」ではない旨をUIに明記、または将来ハッシュ化。
- TTS Workerが匿名＋任意familyCodeを受理＝他家族のElevenLabs課金消費。P0-1の所有者検証と合わせて対処。
- Firebase SDK 10.12.2（約2年前）を11.x系へ更新検討。

### 品質（複数人保守の前提）
- **CIにChromeが無く e2e/画面回帰がCIでスキップ**。`.github/workflows/ci.yml` にChromeセットアップを追加し、`e2e-smoke.js`・`preview-screens.js` をCIで走らせて画面破壊を本番前に検知。
- 空catch 324個・`console.error` 0＝失敗が不可視。少なくともロジック本体のcatchは `_errRecord` へ送るなど可視化。
- cloud-syncのマージ規則が13関数に散在＋キー追加で4-5箇所。スキーマ駆動（キー→マージ戦略の宣言表）へ段階リファクタ。

### UX/学習効果（継続率と対象拡大）
- **保護者の時間制限/休憩タイマー**が皆無（ガチャ×低年齢で最優先の安全機能）。
- **一人で始められない**：既定学年=中1、学年変更がPINの奥のprompt。子どもが絵で学年を選ぶオンボーディング＋1回だけの遊び方ガイドを。
- **ストリークが1日欠けで即0**＝離脱要因。「おやすみチケット（週1回維持）」を導入し、多層の毎日ノルマを整理。
- 小1〜3のコンテンツが手薄（実態は中学＋小学校高学年向け）。偏差値表示に「校正前/幅」の但し書き。

### 権利・ブランド
- アプリ名「小中学生学習アプリ」は一般名詞＝固有ブランド名へ（商標調査）。

## P2：事業として成立に必要
- **無料公開＋ソース全公開のPWAを有料化**する差別化（ネイティブ機能・サポート・独自コンテンツ）。競合＝スタサプ/すらら/RISUは企業。最大の壁は価格でなく「発見されないこと」。
- **Spark無料枠 vs スケール**の矛盾解消（有料顧客を抱えると読5万/書2万/日で同期停止）。P0-1のテナント分離＋メンバーdoc差分書き込み＋ハートビート頻度削減とセットで。
- **将来IAP**：子ども×有料ガチャは最も規制が重い（景表法コンプガチャ・未成年者取消・ベルギー禁止等）。**課金はサブスク/買い切りに限定し、ガチャは虚通貨据え置き**が安全。
- パフォーマンス：3Dの性能ベース自動降格＋`dispose()`実装（GPUリーク）、初回2.3MB JSの分割/遅延、Service Worker再有効化（オフライン）、RUM計測の導入。
