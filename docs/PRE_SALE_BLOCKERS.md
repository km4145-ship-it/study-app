# 販売前ブロッカーと引き継ぎ（2026-07-16 監査より）

5観点（セキュリティ・コード品質・UX/学習効果・商用/法務・パフォーマンス）の徹底監査に基づく、
**販売可能にするまでに必須の作業リスト**。P0＝これが無いと申請不可/法令違反、P1＝審査・権利で高確率に問題化、P2＝事業として成立に必要。

## 現状の結論
**今は販売できるレベルではない。** 技術基盤・学習科学の実装・ガチャ確率開示・広告/分析ゼロ・自作コンテンツは称賛に値するが、
テナント分離・法的文書・アカウント削除・スケール設計が未達。最短でもP0を全て満たすまで申請不可。

## P0：販売前に必須（未対応）

### P0-1【最重要】テナント分離＝匿名 `families/{code}` アクセスの封鎖
**問題**：既定で全ユーザーが匿名認証＋単一doc `families/0000` を共有。`firestore.rules` が `request.auth != null`（匿名でも真）のみのため、
**家族コードを知る/推測する誰でも他家族の子どものPII（氏名・学年・偏差値・学習履歴）を読み書き・全消去できる**。
同時に、grow-onlyの `mu_users` が単一docに無限追記され1MB上限で全員書込停止（スケール不能）。
**セキュリティ・スケール・法務の3ブロッカーが同一原因。**

**解決の方向**：Phase 4の `accounts/{uid}` モデル（実装済み・ルールも堅牢）を既定の稼働パスに昇格し、匿名familiesを閉じる。ただし**後戻り困難＋あなたの手作業が必要**なので、以下を順に：
1. **移行UI**を用意（families/0000のローカルデータ→新規accountへ引き継ぐ導線）。現状のアカウント作成は「空から開始」なので、既存の学習データが引き継がれない（`index.html` アカウント設定の注記どおり）。この引き継ぎを実装しないと既存ユーザーがデータを失う。
2. 既存家族に「アカウント作成へ移行してください」と告知する期間を設ける。
3. 移行完了後、**Firebase Consoleで `firestore.rules` を更新**（このサンドボックスからは `firebase login` 不可＝手作業）。families配下を「非匿名かつ所有者のみ」or 完全閉鎖へ。`accounts/{uid}` 側は既に `request.auth.uid == accountId && sign_in_provider != 'anonymous'` で正しい。
4. ルール反映後、REST APIで「匿名で families/0000 が読めないこと」を実地検証（過去のセキュリティ修正と同じ手順）。

⚠️ **私が単独で「accounts既定化＋ルール封鎖」をしなかった理由**：オンボーディングにメール登録必須化という大きなUX変更を伴い、かつ既存の家族データを巻き込む後戻り困難な操作で、あなたの判断とConsole操作が要るため。移行UIの設計から一緒に進めるのが安全。

### P0-2 法的文書（プライバシーポリシー・利用規約・特商法・サポート窓口）
**問題**：リポジトリに一切なし。App Store/Google Playは掲載時にプライバシーポリシーURLとサポートURLが**必須入力**＝無いと申請を完了できない。日本で有料販売なら特定商取引法表記も必要。
**やること**：
- プライバシーポリシー（収集データ：氏名・学年・生年・学習履歴・エラーログ・端末機種／送信先：Firebase・Cloudflare Workers・ElevenLabs・LINE(GAS)／目的／保持・削除／保護者の権利）。
- 利用規約、特商法表記（個人なら住所公開の是非を検討。App/Google経由でも用意が無難）、問い合わせメール/フォーム。
- GitHub Pagesに静的ページとして公開し、アプリ内設定からリンク。

### P0-3 アカウント削除 → ✅ 対応済み（d3b85ef）
- 追加検討：削除前の再認証UI（requires-recent-login時の再ログイン導線）をもう少し親切に。

### P0-4 13歳未満の保護者同意フロー（COPPA/GDPR-K/個人情報保護法）
**問題**：13歳未満のPIIを、同意取得も第三者送信の開示もなく収集。
**やること**：初回に保護者同意画面（データ収集と第三者送信の要点＋同意）。ElevenLabsへ子どもの名前が読み上げで渡り得る点の精査（名前を読み上げ対象から外す等）。同意撤回＝データ削除の導線（P0-3と接続）。

### P0-5 アセットのライセンス
- **BGM**（`assets/bgm/battle.m4a`・`boss.m4a`・`map.m4a`）：出所・利用許諾が未確認。**確認できないものは配布不可**。ロイヤリティフリー音源に差し替え＋出典記録。
- **絵文字**：OS標準絵文字をcanvasに焼く方式は、配布バイナリに焼き込むとフォントライセンスがグレー。**Twemoji（CC-BY）等**への差し替えを検討。

## P1：審査・権利・安全で高確率に問題化

### セキュリティ（コードで対応可能・次セッション候補）
- `escapeHtml`が引用符（`"`）を非エスケープ→同期される `voice_*` 設定値経由でストアドXSS。`js/util.js` の `escapeHtml` に `"`/`'` を追加し、属性値を使う箇所を確認。
- 管理者PIN/家族PINが `btoa`（暗号化でない・共有同期で復元可能）。子どもの誤操作抑止としては可だが「保護」ではない旨をUIに明記、または将来ハッシュ化。
- TTS Workerが匿名＋任意familyCodeを受理＝他家族のElevenLabs課金消費。P0-1の所有者検証と合わせて対処。
- Firebase SDK 10.12.2（約2年前）を11.x系へ更新検討。

### 品質（複数人保守の前提）
- **CIにChromeが無く e2e/画面回帰がCIでスキップ**。`.github/workflows/ci.yml` にChromeセットアップを追加し、`e2e-smoke.js`・`preview-screens.js` をCIで走らせて画面破壊を本番前に検知。
- 空catch 324個・`console.error` 0＝失敗が不可視。少なくともロジック本体のcatchは `_errRecord` へ送るなど可視化。
- cloud-syncのマージ規則が13関数に散在＋キー追加で4-5箇所。スキーマ駆動（キー→マージ戦略の宣言表）へ段階リファクタ。

### UX/学習効果（継続率と対象拡大）
- **保護者の時間制限/休憩タイマー**が皆無（ガチャ×低年齢で最優先の安全機能）。
- **一人で始められない**：既定学年=中1、学年変更がPINの奥のprompt。子どもが絵で学年を選ぶオンボーディング＋1回だけの遊び方ガイドを。
- **ストリークが1日欠けで即0**＝離脱要因。「おやすみチケット（週1回維持）」を導入し、多層の毎日ノルマを整理。
- 小1〜3のコンテンツが手薄（実態は中学＋小学校高学年向け）。偏差値表示に「校正前/幅」の但し書き。

### 権利・ブランド
- アプリ名「小中学生学習アプリ」は一般名詞＝固有ブランド名へ（商標調査）。

## P2：事業として成立に必要
- **無料公開＋ソース全公開のPWAを有料化**する差別化（ネイティブ機能・サポート・独自コンテンツ）。競合＝スタサプ/すらら/RISUは企業。最大の壁は価格でなく「発見されないこと」。
- **Spark無料枠 vs スケール**の矛盾解消（有料顧客を抱えると読5万/書2万/日で同期停止）。P0-1のテナント分離＋メンバーdoc差分書き込み＋ハートビート頻度削減とセットで。
- **将来IAP**：子ども×有料ガチャは最も規制が重い（景表法コンプガチャ・未成年者取消・ベルギー禁止等）。**課金はサブスク/買い切りに限定し、ガチャは虚通貨据え置き**が安全。
- パフォーマンス：3Dの性能ベース自動降格＋`dispose()`実装（GPUリーク）、初回2.3MB JSの分割/遅延、Service Worker再有効化（オフライン）、RUM計測の導入。
